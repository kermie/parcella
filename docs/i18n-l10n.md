# i18n & l10n

Two related but deliberately **separate** systems:

- **i18n** (`app/i18n.py`) -- which language the interface text is shown
  in.
- **l10n** (`app/l10n.py`) -- how numbers, money, and addresses are
  *formatted*, independent of the interface language.

Keeping them apart was a conscious decision, not an oversight -- see
[Architecture Decisions](./architecture-decisions.md) for the reasoning.
This page is the practical reference: how it works today, and how to
extend it.

## i18n: interface language

- One language per installation (a `ClubSetting` with key `"language"`),
  not per user and not browser-detected -- an association's admin picks
  the language once in **Admin -> Settings**, and everyone sees the same
  thing. A shared club-management tool where members can privately see
  different languages would be more confusing than helpful for a small
  volunteer-run association.
- Every UI string lives in `app/translations/<code>.json`, one file per
  language, structured as nested JSON matching the module layout (e.g.
  `work_hours.overview.title`).
- Currently 7 languages: German (`de`), English (`en`), Polish (`pl`),
  Czech (`cs`), Slovak (`sk`), French (`fr`), Dutch (`nl`). Registered in
  `AVAILABLE_LANGUAGES` in `app/i18n.py`.
- **English is the one and only base/authoring language** -- new UI text
  is written in English first, then translated into the other six
  (German, Polish, Czech, Slovak, French, Dutch). This used to be German
  (the project's original audience was German-speaking), but as an
  open-source project meant for adoption anywhere, English is now the
  single source of truth for both new code and new UI text -- there's no
  longer a split between "authoring language" and "runtime fallback
  language" the way there briefly was; they're the same thing now. See
  Architecture Decisions for the history of this change.
- **English is also the runtime fallback and the fresh-install default**
  -- if a key is missing in the selected language (e.g. a brand-new
  module before its Czech translation is ready), the English string is
  shown. A brand-new installation with no `"language"` ClubSetting set
  yet also starts in English (`DEFAULT_LANGUAGE = "en"` in `app/i18n.py`).
- Templates call `{{ t('key.path') }}`; Python code (error messages shown
  in the web UI, not the REST API) calls `t_for(request, "key.path")`.
  Both are registered once, centrally, in `app/templating.py` /
  `app/main.py` -- no per-router setup needed.
- Placeholders use `{name}` syntax, e.g. `"no_active": "No active
  sponsorships in {year}."`, filled via `t('key', year=2026)`.

### Adding a new language

1. Add the language code and its native name to `AVAILABLE_LANGUAGES` in
   `app/i18n.py`, e.g. `"it": "Italiano"`.
2. Create `app/translations/it.json` with **every** key that exists in
   `en.json` (the source of truth) -- same nesting, nothing missing,
   nothing extra. In practice: copy `en.json` as your starting point and
   translate every value.
3. Keep every `{placeholder}` exactly as-is (same variable name) --
   the code fills these in by name, a renamed or missing placeholder
   breaks that specific string silently (falls back to English, but
   worth catching before it ships).
4. Validate before shipping:
   ```python
   import json
   en = json.load(open("app/translations/en.json", encoding="utf-8"))
   it = json.load(open("app/translations/it.json", encoding="utf-8"))
   # flatten both to dotted-key dicts and diff key sets + {placeholder} sets
   ```
   (This project has done this validation for every language added so
   far -- key parity and placeholder parity, both exact.)
5. Restart the app -- `load_translations()` reads all files at startup
   / import time.
6. No code changes needed anywhere else; the language dropdown in
   Admin -> Settings iterates over `AVAILABLE_LANGUAGES` automatically.

### Known limitation

There's no per-browser or per-user language override (e.g. a login-page
language picker before authentication) -- this was discussed and
deliberately deferred, since it raises an open design question: should
picking a language on the login screen change the *installation-wide*
setting (affecting everyone), or only that one browser session (via a
cookie, never touching the shared ClubSetting)? Worth resolving before
building it, not while building it.

## l10n: region, currency, address format

Three independent `ClubSettings`, none of them tied to `"language"`:

| Setting | Example values | Drives |
|---|---|---|
| `region` | `de_DE`, `en_GB`, `fr_FR`, `nl_NL`, `pl_PL`, `cs_CZ`, `sk_SK` | number format, address field order |
| `currency` | `EUR`, `GBP`, `PLN`, `CZK` | money symbol + its position |

Both are dropdowns in **Admin -> Settings**, right under the language
picker, each with its own help text (translated into all 7 languages).
Defaults: region `de_DE`, currency `EUR` -- independent of the
`language` default being English; these three settings can be any
combination.

### Numbers and money: Babel

`app/l10n.py` uses the [Babel](https://babel.pocoo.org/) library rather
than hand-rolled formatting. Two Jinja filters, registered once in
`app/templating.py`:

```jinja
{{ amount|money }}          {# e.g. "19,99 €" (de_DE/EUR), "£19.99" (en_GB/GBP) #}
{{ value|number }}          {# e.g. "2.000" (de_DE) vs "2,000" (en_GB) #}
{{ value|number(1) }}       {# one decimal place: "3,5" vs "3.5" #}
```

Both read the current request's region/currency automatically (set by
`l10n_middleware` in `app/main.py`, mirroring how `sprache_middleware`
sets the language). `money` accepts an optional currency override if a
specific value needs a different currency than the installation default.

For `<input>` field hints (e.g. the "€" shown next to an amount field
in a form) where a full formatted value doesn't make sense, there's a
separate global: `{{ currency_symbol() }}`.

Babel handles real per-locale differences that are easy to get wrong by
hand -- e.g. Dutch puts the currency symbol *before* the number
(`€ 1.234,50`), French uses a narrow no-break space as the thousands
separator and a non-breaking space before the symbol (`3 000,00 €`),
British English has no space between symbol and number (`£127.50`).

### Address format

Address field order is the one thing about addresses that genuinely
varies among the 7 currently supported countries: postcode-before-city
on its own line (continental Europe) vs. postcode-after-city on its own
last line (UK). `ADDRESS_FORMATS` in `app/l10n.py` is a small dict of
region -> template string covering exactly that:

```jinja
{% set lines = address_lines(member.street, member.postal_code, member.city) %}
{% if lines %}{{ lines|join('<br>')|safe }}{% else %}–{% endif %}
```

`address_lines()` returns a list with empty lines already dropped (e.g.
if postal code and city are both blank). There's also a plain-string
`address()` global if a single string (not a list of lines) is more
convenient for a given template.

**Deliberately not using a full address-format library** (e.g. Google's
`libaddressinput`/`i18n-address` data) -- for the 7 supported countries
there's only one axis of variation, so a five-line dict covers it
without an extra dependency. If a country with a genuinely different
address *structure* (not just postcode position) gets added later,
revisit this.

### Adding a new region

1. Add the locale code and display name to `AVAILABLE_REGIONS` in
   `app/l10n.py`, e.g. `"it_IT": "Italy"`. Must be a locale string Babel
   recognizes (test with `babel.numbers.format_decimal(1234.5,
   locale="it_IT")` first).
2. Add an entry to `ADDRESS_FORMATS` if the country's postcode position
   differs from the continental default (most will just reuse
   `"{street}\n{postal_code} {city}"`).
3. No translation keys needed for the region name itself (region/currency
   names are shown in English on purpose, to avoid needing 7x7 = 49
   extra translated country names for a 2-word dropdown label).

### Adding a new currency

Add the ISO code and a display label (code + symbol, e.g. `"Swiss Franc
(CHF)"`) to `AVAILABLE_CURRENCIES` in `app/l10n.py`. Babel already knows
how to format any real ISO 4217 currency code for any locale -- no
formatting logic to write.

## What still needs money/address formatting applied

If you add a template with a monetary value or a displayed (not
editable-form) address, use `|money`, `|number`, or `address_lines()`
from the start rather than hardcoding a `€` or concatenating
`street`/`postal_code`/`city` by hand -- that hardcoding is exactly what
had crept into 31 places across the app before this was retrofitted
(including a `.replace(",", ".")` hack in the dashboard that only worked
for German).
