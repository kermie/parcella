DELIVERY NOTES -- Community calendar: only STANDARD work sessions
======================================================================

Small, focused change confirming the scope we discussed: the
community calendar already merges work sessions in at read time
(nothing duplicated into a second table -- that was a deliberate
earlier design decision, see docs/module-calendar.md). What changed is
which sessions get merged in.

## What changed

Both places that build the community calendar now filter to
`WorkSession.type == SessionType.STANDARD`:
- The list view (`/calendar/community`)
- The public ICS feed (`/calendar/community.ics`)

SPECIAL (spontaneous/unplanned, e.g. "paint the garden bench today")
sessions no longer appear on the community calendar in either place.
They're untouched everywhere else -- Work Hours itself, signups,
hour-tracking, all unaffected. This is purely about what shows up on
the calendar.

No schema change, no migration.

## Testing

One new test confirms both a STANDARD session shows up and a SPECIAL
one doesn't, in both the list view and the public ICS feed. 68/68
across the whole suite, all passing.

## To apply

1. Unzip into your repo root.
2. `docker compose up -d --build web` (or a plain restart is enough
   this time -- no dependency or Dockerfile change).
3. No migration to run.
